Every token your coding agent emits costs three things: money, latency, and carbon. And here's the uncomfortable part — most of those tokens didn't need to exist.

An agent writes a 12-line helper where a one-liner from the standard library would do. It opens with "Great question! Let me walk you through this…" and closes with "Hope that helps!" It hands a sub-agent a pretty-printed JSON blob that's more than half whitespace. None of that makes the work more correct — all of it shows up on the invoice. In agentic loops, where an agent emits code plus prose plus inter-agent messages at every step, it compounds into the bulk of your bill.

Honey (I Shrunk the AI) removes the waste — reflexively, on every response — while keeping everything load-bearing exact. This is the long read: the philosophy, the three levers, the benchmark numbers (with the honest caveats), and exactly how to install it.

The premise: volume is cost, most volume is waste

Two skills came before Honey and each picked one half. Ponytail made agents write minimal code. Caveman made them write terse prose. Honey combines both, adds a third lever the others don't have, and — crucially — a set of safety carve-outs so "minimal" never means "broken."

Write the minimum code that needs to exist, and say it in the fewest words that stay clear — but keep everything load-bearing exact.

The hard part isn't "write less." Anyone can truncate. The hard part is knowing the difference between waste and substance — and Honey encodes that difference rather than leaving it to vibes.

The three levers

1 Less code

Walk a ladder — needs to exist? → stdlib → language native → existing dependency → one line → minimum block — and stop at the first rung that works. The cheapest line is the one you never write.

2 Less prose

Drop the wind-up, the hedging, the narration of code that already speaks for itself. Fragments over paragraphs. Answer first; context only if load-bearing.

3 Denser handoffs

When the reader is another agent, hand it the most token-efficient format it parses losslessly — compact/columnar JSON, or ESO. Cuts handoff size ~in half at zero loss. Fires only here.

Lever 1 — the code ladder, in full

  1. Does it need to exist at all? Often the best move is no code — a config change, an existing call site, or deleting the requirement.
  2. Standard library — don't hand-roll what itertools, pathlib, collections, or datetime already do.
  3. Language native — an operator, a comprehension, a dict lookup over an if-ladder.
  4. An installed dependency — use what the project has; don't add one for four lines.
  5. One line before reaching for a block.
  6. Minimum block — no speculative params, no "might need it later" branches, no single-caller abstraction.

Speculative generality is the single costliest agent habit. Code for an imagined requirement is pure overhead, and the requirement usually never arrives.

Lever 2 — say less about it

Most words around code are filler. The reader wanted the answer, not the throat-clearing.

  • Drop the wind-up & wind-down — no "Great question!", no "hope this helps!", no restating your prompt back.
  • Drop the hedging — "use X," not "you might possibly consider perhaps X." State real uncertainty once, briefly.
  • Don't narrate readable code — explain the why and the non-obvious; skip the what the code already says.
  • Answer first. Context only if load-bearing.

One caveat the lever never crosses: never compress the things you'd copy, paste, or run. Prose gets terse; precision stays exact.

Lever 3 — ESO, the agent-to-agent format

Pretty-printing JSON is roughly +55% tokens for nothing. ESO (Efficient Structured Output) is a zero-dependency, schema-first format where repeated record keys are emitted once and declared row counts catch truncated messages:

printf '%s' '{"from":"reviewer","findings":[{"sev":"H","issue":"expired token"}]}' | eso encode
eso decode < handoff.eso

What Honey refuses to cut

This is what makes it safe for production rather than just clever. Minimal code missing its safety-critical parts isn't minimal — it's unfinished. Honey never compresses away:

  • Input validation at trust boundaries — user input, network, files, env.
  • Error handling that prevents data loss or corruption.
  • Security — auth checks, escaping, secrets handling.
  • Accessibility basics — labels, roles, keyboard paths.
  • Visual & UX polish when the deliverable is user-facing — for landing pages and UI, polish is the requirement, not bloat.
  • Anything you explicitly asked for.

And it never touches what you'd copy, paste, or run. Code blocks stay verbatim; identifiers, paths, commands, versions, error messages — exact. "The auth middleware" ≠ requireAuth(). The rule: if compressing makes the reader work to recover the meaning, you moved cost, you didn't remove it. Lazy ≠ broken.

Auto-intensity — it tunes itself

Honey isn't one setting. It reads intensity from your phrasing, with no deliberation tax:

  • lite — "explain," "why," "should I." The explanation is the deliverable, so prose stays.
  • full — "write," "fix," "build." Terse, fragments, ship it. The default.
  • ultra — "just," "quick," "one-liner." Answer-only — but still naming the one edge case that bites (raises KeyError on a missing key — use .get).

It steps up a mode, never down, when terseness would drop correctness. Pin it manually: /honey lite|full|ultra, or /honey off.

Does it actually work? The benchmark

Honey ships a reproducible benchmark: 23 tasks across three kinds of work — baseline vs Caveman vs Ponytail vs Honey — same model, same prompts, only the skill changes. Correctness is objective (unit tests, structural/a11y checks, lossless round-trip recovery). Quality is a 4-model cross-family judge panel (Opus 4.8 + Sonnet 4.6 + Haiku 4.5 + GPT-5.5) under a neutral rubric that says nothing about length — so a terse skill gets no thumb on the scale.

Quality is % of baseline; tokens are generated output vs baseline:

  • Code — the deepest cut (−49%) at tied quality (98% vs 100%, within judge noise). Ponytail's mandatory self-check inflates trivial code by +24%.
  • User-facing — the carve-out keeps Honey from compressing polish, yet it still trims (−6%) while earning the top quality score and the only 100% a11y pass.
  • Agent-to-agent — under adversarial relay queries Honey is the only variant that stays 100% lossless while halving handoff size. Its cleanest win.

The headline isn't "saves the most tokens" — crude tricks save more by getting things wrong. It's saves tokens while staying correct. The same pattern holds on GPT-5.5.

A note on honesty

Honey's repo corrects itself. Earlier README versions quoted flashier numbers (92/78/73% quality) from an unpublished run; they don't reproduce, so the committed harness's narrower table replaced them. And the input-compression lever ships as a measured negative result: −16.5% on a verbose corpus, but only 2.5% (median 0%) on 266 real human prompts — because real prompts are already terse. It's documented as a negative, not buried. That's the spirit of the whole project: don't overstate.

Why this matters

Agentic coding is where the savings stop being a rounding error. An agent loop generates enormous volume — code, prose about code, messages between sub-agents — at every step. Multiply a 30–50% output reduction across thousands of turns and it's the difference between a workflow that's economical and one that isn't.

There's a quieter benefit, too. Terser output is often clearer output. Strip the hedging and the narration and what's left is the decision, the code, and the one caveat that matters. Less to read, less to misread.

Honey doesn't tell your agent to "write less." It teaches it to remove the waste and keep the substance — and ships the benchmark, the carve-outs, and even a negative result so you can check that it does.
